## 1. システム全体概要
本システムは、PostgreSQL を「唯一の真実のソース」とし、**Transactional Outbox パターン**と **LISTEN / NOTIFY** を活用して、低負荷かつ高信頼なメッセージ配信を実現する。



---

## 2. バックエンド：データ保存・採番要件

### 2.1 原子的な書き込み (Atomic Write)
* **トランザクション管理**: メインデータ（Message等）の保存、スコープ別の採番、および配信フィード（Outbox）の保存は、同一のデータベーストランザクション内で実行しなければならない。

### 2.2 分散スコープ採番 (Sharded Sequencing)
* **採番単位**: 「ルーム単位（ROOM）」または「ユーザー単位（USER）」で独立した連番を発行する。
* **厳密連番の維持**: 各スコープ内において、ID は欠番のない連続した整数であることを保証する。
* **ドメイン整合性**: `DeliveryFeed` エンティティは生成時点で有効な連番 ID を保持し、ドメイン層の不変条件を満たすこと。

### 2.3 配信フィード生成 (Delivery Feed Factory)
* **DTO 変換**: `DeliveryFeedFactory` はドメイン層に配置し、エンティティを配信専用の JSON ペイロードへ変換する。
* **型安全スキーマ**: ペイロードは `event_type` をタグとした「識別子付きユニオン体」形式とし、クライアント側での型安全なパースを支援する。

---

## 3. バックエンド：配信・リレー要件

### 3.1 リアルタイム通知 (LISTEN / NOTIFY)
* **DB トリガー**: `delivery_feeds` への挿入時に `pg_notify` を実行し、リレープロセスへ即時合図を送る。
* **低負荷待機**: リレープロセスは `psycopg` 等を用いた非同期 `LISTEN` で待機し、不要なポーリングクエリを排除する。

### 3.2 確実なリレー (External Relay)
* **FastAPI 組み込み**: リレー処理は FastAPI の `Lifespan` タスクとしてバックグラウンドで実行する。
* **重複排除配信**: 複数インスタンス実行時は `FOR UPDATE SKIP LOCKED` を用い、同一フィードの重複送信を防止する。
* **セーフティネット**: 通知漏れに備え、定期的（例: 1分間隔）な未送信レコードのスキャンを併用する。

---

## 4. フロントエンド：ロスト検知・復旧要件

### 4.1 ロスト検知（Gap Detection）
* **連番監視**: 受信した `delivery_id` が `保持している最新ID + 1` でない場合、即座にデータロスト（または順序逆転）と判断する。


### 4.2 べき等なデータ操作 (Idempotency)
* **ID ベースの上書き**: `payload.id`（ビジネス ID）を用いてリスト内のデータを特定し、既存データがある場合は「追加」ではなく「更新（上書き）」を行うことで、二重配信による重複表示を防ぐ。

### 4.3 自動同期 (Catch-up / Resync)
* **差分取得**: ロスト検知時、または再接続時に、自身の `last_seen_id` を用いてサーバーから未受信分を一括取得する。

---

## 5. 運用・パフォーマンス要件

### 5.1 パフォーマンス最適化
* **部分インデックス**: `WHERE status = 'PENDING'` を対象とした **Partial Index** を作成し、送信済みレコードが増大してもリレー速度を低下させない。
* **コネクション分離**: `LISTEN` 用には専用の DB 接続を確保し、API のリクエスト用接続プールを圧迫しない。

### 5.2 データ・メンテナンス
* **ログ掃除**: 配信完了後一定期間が経過した `delivery_feeds` レコードは、アーカイブ化または物理削除を行うバッチ処理を検討する。

---

## 仕様のまとめ（設計のポイント）

| 特徴 | 実装手段 | もたらすメリット |
| :--- | :--- | :--- |
| **ロストゼロ** | Transactional Outbox | DB にあれば必ずいつか届く。 |
| **順序保証** | 厳密連番（スコープ別） | クライアントが「抜け」を 100% 検知できる。 |
| **超低負荷** | LISTEN / NOTIFY | メッセージがない時の CPU/DB 負荷がほぼ 0。 |
| **高可用性** | SKIP LOCKED | サーバーを何台並べても安全に配信が分散される。 |
