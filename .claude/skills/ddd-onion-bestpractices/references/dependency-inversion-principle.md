# Dependency Inversion Principle (依存性逆転の原則)

## 1. 責務と制約

### 責務
*   **ビジネスロジックの保護**: 上位モジュール（ドメイン・アプリケーション層）を、下位モジュール（DB、外部API、フレームワーク）の変更から隔離する。
*   **詳細からの独立**: 「どのようにデータを保存するか（SQL, NoSQL等）」ではなく、「何を保存する必要があるか」という抽象的な要求に基づいてシステムを設計する。
*   **交換可能性の確保**: インターフェースを介することで、技術スタック（DBの変更やモックへの差し替え）をビジネスロジックに触れずに変更可能にする。

### 制約
*   **依存の方向**: 依存関係の矢印は常に「外側から内側」に向ける。内側の層は、外側の層で定義された具体的なクラスをインポートしてはならない。
*   **抽象への依存**: 上位モジュールは具体的な実装（Concrete Class）ではなく、必ず抽象（Interface / Abstract Base Class）に依存する。
*   **インターフェースの所有権**: インターフェースは、それを利用する側（上位モジュール）と同じ層に定義する。これを「プラグイン・アーキテクチャ」の考え方で運用する。



---

## 2. 構成要素

### 1. Abstraction (インターフェース)
**定義:** ドメイン層またはアプリケーション層で定義される、「何をするか」のみを記述した抽象クラス。
*   **実装例**: Python の `abc.ABC` や `typing.Protocol` を使用した抽象基底クラス。
*   **配置**: `domain/repositories/` や `application/interfaces/`。

### 2. Low-level Module (具象実装)
**定義:** 抽象インターフェースを実際に実装したクラス。
*   **特徴**: SQLAlchemy、Boto3、httpx などの外部ライブラリを使い、技術的な詳細を記述する。
*   **配置**: `infrastructure/` 以下の各ディレクトリ。

### 3. Dependency Injection (注入)
**定義:** 上位モジュールが要求するインターフェースに対し、外部（Composition Root）から具象実装を流し込む仕組み。
*   **役割**: クラス内部で `new` (インスタンス化) を行わず、外部から「提供」されることで、テスト時にモックへの差し替えを容易にする。

### 4. Composition Root (構成の根幹)
**定義:** アプリケーションの起動時に、どのインターフェースにどの実装を割り当てるかを設定する場所。
*   **FastAPIでの役割**: `dependencies.py` や各 Router の `Depends()` がこの役割を担い、インフラ層の具象クラスをアプリケーション層のユースケースに注入する。

---

## 3. 逆転のイメージ（before/after）

*   **DIPなし（従来）**:
    `Application` $\rightarrow$ `SQLAlchemyRepository` (上位が下位の詳細に依存)
    *詳細が変わると、ビジネスロジックも修正が必要になる。*

*   **DIPあり（オニオン）**:
    `Application` $\rightarrow$ `IRepository (Interface)` $\leftarrow$ `SQLAlchemyRepository`
    *上位と下位の両方が、中央にある「抽象」に依存する。これにより詳細を自由に交換できる。*
