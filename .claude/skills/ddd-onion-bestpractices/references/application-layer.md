# Application Layer (アプリケーション層)

ソフトウェアが「何ができるのか（ユースケース）」を実現するために、ドメインオブジェクトを調整（オーケストレーション）する層です。

## 1. 責務と制約

### 責務
*   **ユースケースの実現**: ユーザーや外部システムからの要求（「注文する」「メッセージを送る」など）を進行する。
*   **オーケストレーション**: ドメイン層のオブジェクト（Entity, VO, Domain Service）を適切な順序で呼び出す。
*   **トランザクション管理**: データベースのコミットやロールバックの境界を決定する。

### 制約
*   **ドメイン層にのみ依存する**: インフラ層（DBの実装詳細）やプレゼンテーション層（HTTPリクエスト等）に依存してはならない。
*   **ビジネスロジックを持たない**: 計算、状態の正当性判定、ルールの検証などはすべてドメイン層の責務。アプリケーション層は「処理の進行」にのみ徹する。
*   **技術の隠蔽**: データの保存や取得は、必ずドメイン層で定義された Repository インターフェースを介して行う。

---

## 2. 構成要素

### 1. Application Service (ユースケース)
**定義:** 具体的なユースケース（シナリオ）をメソッドとして実装するクラス。
*   **特徴**: メソッド一つが、システムに対する一つのトランザクションとなる。
*   **入出力**: プレゼンテーション層からは、プリミティブな値や DTO（後述）を受け取り、結果も DTO などを返す。

### 2. DTO (Data Transfer Object)
**定義:** 層の間でデータを渡すための、ロジックを持たない純粋なデータ構造。
*   **役割**: ドメインオブジェクト（エンティティ等）の内部構造を、直接プレゼンテーション層（APIのレスポンス等）に漏らさないためのクッション。
*   **特徴**: メソッド（振る舞い）を持たない。

---

## 3. アプリケーションサービスの典型的なフロー

DDD におけるアプリケーションサービスのメソッドは、一般的に以下の 4 つのステップで構成されます。

```python
async def update_request_status(self, request_id: int, new_status: str, operator: str) -> RequestDTO:
    # 1. 準備・再構築: リポジトリから既存のエンティティを取得する、または新規作成する
    request = await self._repo.get_by_id(request_id)
    if not request:
        raise EntityNotFoundException()

    # 2. 実行: エンティティのドメインロジック（振る舞い）を呼び出す
    status_vo = RequestStatus(new_status)
    request.transition_to(status_vo) # ※ここが重要。状態変更ロジックはエンティティに任せる

    # 3. 永続化: 変更されたエンティティをリポジトリを通じて保存する
    saved_request = await self._repo.save(request)

    # 4. 外部への通知・返却: イベントのパブリッシュや、DTOへの変換を行う
    await self._event_publisher.publish(...)
    return RequestDTO.from_entity(saved_request)
```

## 4. ドメインサービスとの違い

名前が似ていますが、明確に責務が異なります。

| 特徴 | Application Service (アプリケーション層) | Domain Service (ドメイン層) |
| :--- | :--- | :--- |
| **役割** | ユーザーの「やりたいこと（ユースケース）」を進行する | 複数のエンティティにまたがる「ビジネスルール」を表現する |
| **依存** | リポジトリ（インターフェース）や外部サービスを呼び出す | 外部には一切依存せず、ドメインオブジェクトのみを扱う |
| **例** | `CheckoutService` (カート取得→計算→DB保存→通知) | `TaxCalculator` (複数商品の税率から総額を算出するロジック) |
