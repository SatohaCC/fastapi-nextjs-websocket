### 💡 なぜReactコンポーネントは「純粋」でなければならないのか？

Reactは、コンポーネントが**同じ入力（props/context）なら常に同じ出力（JSX）を返す「純粋関数」である**と仮定して設計されています。不純なコードを書くと、以下のような問題が発生するためです。

* 画面表示が意図せずバグる。
* React Compiler（最適化）や新機能（Suspense、Concurrent Renderingなど）が導入された際に急に壊れる。
* コードの可読性や保守性が低下する。

---

### ❌ 守るべき「5つのべからず」（不純なコードの例）

1. **レンダー中にコンポーネント外部の変数を書き換えるべからず**
* （例：関数外で定義した `let renderCount = 0` をレンダー中に `renderCount++` する）
* **理由:** Reactがコンポーネントをいつ、何回実行するかは保証されていないため、表示や数値がバグる原因になります。


2. **レンダー中に外部（localStorageやグローバル変数）から直接データを読み取るべからず**
* （例：`localStorage.getItem("userId")` を直接JSXの評価に使う）
* **理由:** 外部の状態が変わってもReactが検知できず、画面が古いままになったり、意図しないタイミングで表示が変わったりします。


3. **同じ入力（props）なのに、実行のたびに結果が変わる処理を入れるべからず**
* （例：レンダー中に `Math.random()` や `new Date()` を直接使って条件分岐する）
* **理由:** Reactが画面を再描画（再計算）するたびに結果がブレてしまい、UIの一貫性が壊れます。


4. **レンダー中にネットワークリクエスト（APIコール）を行うべからず**
* （例：関数内で直接 `fetch('/api/log')` を呼び出す）
* **理由:** 再レンダーのたびに無駄なリクエストが大量に走り、サーバーやパフォーマンスに悪影響を与えます。


5. **本番コードのレンダー中に `console.log` を残すべからず**
* **理由:** デバッグ時の一時的な使用は問題ありませんが、ログ出力も立派な「外部への書き込み（副作用）」であるため、不要になったら必ず削除すべきです。



---

### ⭕ 良いコードを書くための「4つの鉄則」

1. **外部への書き込みや読み込みは、必ず `useEffect` か「イベントハンドラ」の中に閉じ込めるべからず**
* API通信、localStorageの操作、DOMの直接操作などは、レンダーのプロセス（JSXを返す処理）の外側で実行します。


2. **レンダー中の「ローカルな書き換え」は安全に活用するべからず**
* 関数コンポーネントの**内部でその都度生成された変数や配列**であれば、その中で自由に `push` したり書き換えたりしても、外部に影響を与えない（純粋性を保てる）ため問題ありません。


3. **副作用が必要な場合は、適切なデータフロー（`useState` やデータフェッチライブラリ）に変換するべからず**
* 外部からデータを取得したい場合は、直接読み込むのではなく、状態（State）として管理するか、信頼できるデータフェッチ用のフレームワーク・フックを経由させます。


4. **「Reactが関数を何度実行しても問題ない設計」を常に意識するべからず**
* 1回の表示に対してコンポーネント関数が2回、3回と実行されても、画面の見た目やアプリのデータが絶対に壊れないような、堅牢で決定論的なコードを書きます。
