
## 1. 配信基盤の全体概要
本システムは、ビジネスロジックの実行と同時に「配信フィード（Outbox）」をDB内に記録し、それをバックグラウンドワーカーが順次読み取って配信するアーキテクチャを採用する。



---

## 2. 機能要件 (Functional Requirements)

### 2.1 厳密連番の採番 (Strict Sequencing)
*   **欠番の禁止**: 標準の `SERIAL` 等は使用せず、採番専用テーブル（カウンタテーブル）を用いた更新処理により、欠番のない連続した整数 ID を発行すること。
*   **原子性の保証**: エンティティの保存、採番、および配信フィード（Outbox）への書き込みは、必ず同一のデータベーストランザクション内で完了しなければならない。

### 2.2 配信フィード生成 (Delivery Feed Factory)
*   **スキーマ変換**: `DeliveryFeedFactory` を通じて、ドメインエンティティ（Message 等）を配信用 JSON（Payload）に変換すること。
*   **タグ付きユニオン体**: ペイロードは `event_type` に応じて構造を完全に切り替え、クライアント側で型安全にパース可能な形式とすること。

### 2.3 組み込み型外部リレー (FastAPI Internal Relay)
*   **バックグラウンド実行**: FastAPI の `Lifespan` イベントを利用し、アプリケーションの起動と同時にリレーワーカーを常駐させること。
*   **非同期ポーリング**: ワーカーは一定間隔（100ms以下）で DB を監視し、未送信のフィードを Redis PubSub へパブリッシュすること。
*   **重複排除（SKIP LOCKED）**: 複数インスタンス稼働時でも、`FOR UPDATE SKIP LOCKED` を用いて同一レコードの二重送信を防止すること。

### 2.4 ロスト検知とリカバリ (Recovery)
*   **クライアント側検知**: クライアントは受信した ID が `last_id + 1` でない場合、即座にロストと判断すること。
*   **差分取得インターフェース**: 指定した ID 以降のフィードを `delivery_feeds` テーブルから一括取得できる API を提供すること。

---

## 3. 非機能要件 (Non-Functional Requirements)

### 3.1 整合性 (Consistency)
*   **At-least-once 配信**: ネットワーク障害等で配信に失敗しても、DB に `PENDING` が残る限りリトライされ、最低1回は必ず配信されること。
*   **順序の絶対保証**: DB に記録された `id` の昇順が、そのまま Redis/WebSocket での配信順序と一致すること。

### 3.2 運用・保守性
*   **配信ログの永続化**: `delivery_feeds` は単なるキューではなく、永続的な配信ログとして保持し、過去の任意の時点からの同期を可能にすること。
*   **監視**: 配信遅延（未送信レコードの滞留）を検知し、アラートを発報できる状態を維持すること。

## 4. 技術仕様 (Technical Specifications)

### 4.1 テーブル構造
| テーブル | 役割 | 主要カラム |
| :--- | :--- | :--- |
| `delivery_sequences` | 厳密採番 | `name` (PK), `last_id` |
| `delivery_feeds` | アウトボックス兼ログ | `id` (PK/厳密連番), `event_type`, `payload` (JSONB), `status` (PENDING/PUBLISHED) |

### 4.2 配信フロー (The Outbox Loop)
1.  **[API]** トランザクション開始
2.  **[API]** メインデータ保存 ＋ カウンタ `+1` ＋ フィード保存（`PENDING`）
3.  **[API]** コミット
4.  **[Relay]** `SELECT ... FOR UPDATE SKIP LOCKED` で `PENDING` を取得
5.  **[Relay]** Redis へ `PUBLISH`
6.  **[Relay]** `status` を `PUBLISHED` に更新してコミット

---

## 5. 原則との整合性に関する定義

*   **エンティティの完全性**: `DeliveryFeed` エンティティは、インフラ層のリポジトリで採番が行われた瞬間に生成される。これにより、ドメイン層に渡る時点ですべての不変条件（特に ID の存在）を満たした「正しい状態」であることを保証する。
*   **責務の分離**: エンティティはデータの正しさに責任を持ち、Factory は外部への見せ方に責任を持ち、Relay は到達性に責任を持つ。
