# スケーラビリティ評価レポート

現在の実装を確認した結果、**中規模（数千〜数万のアクティブ接続）までは水平スケーリング可能な堅牢なアーキテクチャ**となっていますが、さらなる大規模化に向けてはいくつかのボトルネックが存在します。

## 1. アーキテクチャの強み（スケーラビリティに貢献している点）

### 水平分散への対応
- **Redis Pub/Sub による状態同期**: メッセージの配信に Redis を介しているため、バックエンドの Pod 数を増やしても、全ユーザーにリアルタイムでイベントが届く設計になっています。
- **ステートレスな設計**: ユーザーの接続状態はメモリ（`ChatManager`）で持ちつつ、配信ロジックは Redis 経由で行われるため、どのサーバーに接続しても一貫した体験が得られます。

### 信頼性と整合性の確保
- **Transactional Outbox パターン**: メッセージの保存とイベント発行を同一トランザクションで行い、`relay_worker` が非同期で配信するため、DB更新とRedis通知の不一致（一方が失敗するなど）を防いでいます。
- **厳密連番（Sequence ID）**: クライアントが最後に受け取った ID を保持し、再接続時にギャップを自動補完できるため、パケットロスやサーバー切断に非常に強い設計です。

### 同時実行制御
- **FOR UPDATE SKIP LOCKED**: `relay_worker` が未配信レコードを取得する際、ロックされた行をスキップするため、複数のワーカー（複数Pod）が同時に動いても二重配信が発生しません。

## 2. 潜在的なボトルネックと改善提案

### A. DB 採番の直列化（最大のボトルネック）
- **現状**: `delivery_sequences` テーブルの特定の 1 行を毎回更新して ID を採番しています。
- **影響**: 書き込みスループットが Postgres の単一行更新性能に制限されます。
- **改善**:
    - 高負荷時は Snowflake ID や UUID v7 などの分散採番を採用する。
    - ただし、現在の「厳密連番によるギャップ検知」のメリットを維持する場合、シャーディング（ユーザーごと、またはルームごとにシーケンスを分ける）を検討してください。

### B. Outbox テーブルの肥大化
- **現状**: `delivery_feeds` テーブルに全配信履歴が蓄積され続けています。
- **影響**: インデックスの肥大化、クエリ速度の低下、ストレージ圧迫。
- **改善**:
    - 配信済み（`PUBLISHED`）かつ一定期間経過したレコードをアーカイブまたは削除するクリーンアップジョブを追加する。

### C. 全 Pod へのブロードキャスト負荷
- **現状**: 1 つの Redis チャンネルを全 Pod が購読し、自分に関係ないメッセージも一度パースして捨てています。
- **影響**: ユーザー数と Pod 数が極端に増えると、Redis 購読処理の CPU 負荷が無視できなくなります。
- **改善**:
    - チャンネルをハッシュ（例: `chat:room_1`, `user:alice`）で分割し、Pod はその時必要なチャンネルのみを購読するようにする。

### D. クライアント側のメモリ管理
- **現状**: `useState` で全メッセージ履歴を保持しています。
- **影響**: 数千件のメッセージが溜まると React の再レンダリングが重くなり、メモリを消費します。
- **改善**:
    - バーチャルリスト（Windowing）の採用。
    - 古いメッセージをメモリからパージし、必要に応じて REST API で再取得する。

---

## 3. 次のステップへの推奨事項

1. **クリーンアップジョブの実装**: `delivery_feeds` の整理ロジックを追加。
2. **Redis チャンネルの細分化**: 大規模化が見込まれる場合、全配信ではなくトピックごとの購読へ移行。
3. **負荷テストの実施**: Locust や k6 を使用し、特に `delivery_sequences` のロック競合がどのレベルで発生するかを計測する。

全体として、現在の設計は**非常にクリーンで拡張性が高い**です。上記のリスクを把握していれば、フェーズに合わせて段階的にスケールアップ可能です。
