# Pub/Sub におけるメッセージロストの検知と再同期パターン

リアルタイム通信システムでは、配信経路のどこかでメッセージが失われる可能性があります。何が原因でロストし、どう検知するかをパターン別に分類します。

---

## 1. ロストが発生する主な箇所

| 発生箇所 | 原因の例 | 影響 |
| :--- | :--- | :--- |
| **送信側 (Publisher)** | DB保存後のブローカー（Redis等）への送信失敗 | 全ての購読者がメッセージを受け取れない |
| **ブローカー内部** | Redisの再起動、メモリ溢れによる古いメッセージの破棄 | 配信待ちのメッセージが消失する |
| **受信側 (Subscriber)** | ネットワーク瞬断、ブラウザのバックグラウンド化、クラッシュ | 特定のユーザーだけがメッセージを受け取れない |

---

## 2. ロスト検知のパターン

### A. シーケンス番号 (Sequence Numbers)
メッセージに連番（1, 2, 3...）を付与し、受信側で「番号の飛び」を確認する方式。
- **検知方法**: `受信した番号 != 前回受信した番号 + 1` であればロストと判断。
- **特徴**: 実装が比較的容易。現在のシステムでもこの一部を採用。

### B. ハートビート / タイムアウト (Heartbeats)
一定間隔で「生存確認メッセージ（Ping/Pong）」を送り、返答がない場合に切断とみなす方式。
- **検知方法**: 最後に通信があった時刻から一定時間経過したら「接続が死んでいる」と判断。
- **特徴**: 「ネットワークは繋がっているがパケットが届かない」状態（ハーフオープン）を検知できる。

### C. クライアントサイド・チェックポイント
クライアントが最後に正常処理した ID やタイムスタンプを永続ストレージ（localStorage 等）に保存しておく方式。
- **検知方法**: 再接続時に「最後に持っているのは X 番です」と自己申告し、サーバー側の最新状態と比較する。

---

## 3. 整合性を確保するための設計パターン

ロストを検知した後、どうやって「埋め合わせ」をするかのパターンです。

### ① 投げっぱなし (Fire and Forget)
ロストを許容する設計。
- **用途**: ログ送信、リアルタイムの株価、ゲームの座標更新など。
- **特徴**: 最も高速だが信頼性はない。

### ② ポーリング・バックストップ (Periodic Sync)
Pub/Sub とは別に、定期的に（例: 30秒ごと）HTTP で「差分はありませんか？」と問い合わせる。
- **用途**: Pub/Sub の漏れを補完する補助的な手段。
- **特徴**: 実装が簡単だが、リアルタイム性が低く、サーバー負荷が増える。

### ③ 履歴問い合わせ (Fetch After ID)
ロストを検知した瞬間に、特定の ID 以降のデータを API で取得し直す。
- **用途**: チャット、通知。
- **特徴**: ロストした瞬間だけ動くので効率的。ただし、「更新（Update）」や「削除（Delete）」を扱うには ID だけでは不十分。

### ④ ログベース配信 (Log-based Pub/Sub / Redis Streams)
ブローカー自体が履歴を保持する（Redis Streams や Kafka 等）。
- **用途**: 銀行、高信頼メッセージング。
- **特徴**: 購読者が「105番から再送して」とリクエストすると、ブローカーが過去の履歴を再送してくれる。

### ⑤ トランザクショナル・アウトボックス (Transactional Outbox)
DB の更新と「イベントの記録」を一つのトランザクションで行う。
- **用途**: データの整合性が最優先されるシステム。
- **特徴**: DB に「イベントログ」が残るため、後から確実に全履歴を辿ることができる。

---

## まとめ：どのパターンを選ぶべきか

- **データの欠損が許されない（金銭、重要なステータス）**:
  → **パターン⑤（アウトボックス）** ＋ **パターン④（ログ配信）**
- **最新の状態さえ分かれば良い（プレゼンス表示など）**:
  → **パターン②（定期同期）**
- **追記のみで、速度を優先したい（ログなど）**:
  → **パターンA（シーケンス番号）** ＋ **パターン③（履歴問い合わせ）**
