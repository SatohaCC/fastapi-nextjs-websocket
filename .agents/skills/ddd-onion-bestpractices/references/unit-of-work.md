# Unit Of Work (単位操作)

## 1. 責務と制約

### 責務
*   **アトミックな操作（原子性）の保証**: 1つのユースケースにおける複数のデータ更新を「すべて成功」か「すべて失敗」のいずれかに制御し、データの不整合を防ぐ。
*   **トランザクション境界の定義**: ビジネスロジックの区切り（ユースケース）をデータベースのトランザクション単位として明確に定義する。
*   **リポジトリの調整**: 同一トランザクション内で動作する複数のリポジトリに対し、共通のデータベースセッション（Connection）を提供する。

### 制約
*   **アプリケーション層での制御**: トランザクションの開始・確定（Commit）・破棄（Rollback）の判断は、必ず `Application Layer` で行う。
*   **リポジトリでの Commit 禁止**: `Infrastructure Layer`（Repository）内で `commit()` を呼び出してはならない。リポジトリはデータの「状態変更の予約（Flush）」までを担当する。
*   **ドメイン層からの独立**: `Domain Layer` のオブジェクトは UoW の存在を知ってはならず、永続化のタイミングに依存したロジックを持ってはならない。
*   **コンテキストマネージャの利用**: Python の `with` 構文を利用し、例外発生時の自動ロールバックを確実に行う実装とする。

---

## 2. 構成要素

### 1. IUnitOfWork (Interface)
**定義:** トランザクション操作を抽象化したインターフェース。
*   **役割**: `commit()`, `rollback()` メソッド、および Python のコンテキストマネージャ（`__enter__`, `__exit__`）を定義する。
*   **配置**: `application/interfaces/` または `domain/repositories/` (抽象度による)。

### 2. Concrete Unit of Work (Implementation)
**定義:** 特定のデータベース技術（SQLAlchemy, Tortoise ORM 等）に依存した UoW の実体。
*   **役割**: 実際の DB セッションを管理し、リポジトリにそのセッションを注入する。
*   **配置**: `infrastructure/persistence/`。

### 3. Repository Accessor
**定義:** UoW を通じて各リポジトリへアクセスするための仕組み。
*   **役割**: `uow.users` や `uow.messages` のように、同一セッションを共有するリポジトリ群をプロパティとして提供する。

### 4. Transaction Context
**定義:** `with uow:` ブロックによって形成されるスコープ。
*   **役割**: このスコープ内で行われるすべての操作が、一つのデータベーストランザクションとして扱われることを視覚的・構造的に保証する。
